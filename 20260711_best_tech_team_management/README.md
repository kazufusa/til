# リポジトリ運用ルール（たたき台）

マルチAIエージェントによる実在部署の仮想組織化 PoC の共有リポジトリ。
2カ月の短期PoC。最終成果物はWordドキュメント（報告書）。コードは実験物。

## ツールの役割分担

- Word（同時編集）= 報告書 兼 会議の合意アイテム 兼 最終成果物。決定・合意はここに書く
- GitHub（このリポジトリ）= コードと技術詳細。Wordから必要な所をリンクする

決定・合意の真実はWord、コードの真実はGitHub。役割を重ねない。
リポジトリ運用の目的は2つだけ。

- 各自がぶつからずに並行して動ける
- 技術詳細とコードの置き場になる

## ディレクトリ構成

```
/pocs/<poc名>/     PoCごと。担当がこの中で自由にやる
/docs/             技術文書。Word報告書からリンクする先
DECISIONS.md       決定ログ（追記専用）。決定の正本
README.md          このファイル
.gitignore         鍵・実データ・生成物を除外
```

肝は、PoCごとにディレクトリを分けること。各自が自分のディレクトリ内で完結するので、
コンフリクトがほぼ起きない。だから重いブランチ運用が要らない。

## 数字の記録ルール

比較を成立させるための最小限の書式。各PoCの数字が揃わないと比較できないので、これだけ揃える。
報告書テンプレ（報告書_作成中.md）に定義。テックリードが揃える書式であって、評価の設計ではない。

- トークン: システムプロンプトとリトライ分を含める。マルチエージェントはPoC単位で合算
- モデル・実行日を必ず記録（違えば精度もコストも変わる）
- コストはモデル別単価から算出

精度をどう測るか（何を正解とするか）は評価の設計そのもので、正解定義オーナーが報告書4章・5章で定める。書式とは別。

## ブランチ / PR

- 自分のPoCディレクトリ内は、自己判断でマージしてよい。レビューゲートは置かない
- レビューが要るのは共有ファイル（共通の土台コードなど）だけ。ここだけPRで合意を取る
- 共有部分の変更は、Wordで決めた全体MTGの決定と紐づける

独立ディレクトリは自由に、共有部分だけ慎重に。速度と統合を両立させる。

## 案と決定の流れ

- 技術的な案 = GitHub（PR/Issue）で先に出しておく
- 軽い案 = Wordに直接書いてその場で決める
- 決定の正本は GitHub の `DECISIONS.md`（追記専用）。1行ずつ積む: 日付・論点・決定・異論の有無/サインオフ
  - Wordは合意の場・成果物だが、同時編集は上書きで消えやすく追記ログの保管に向かない。監査できる履歴はgit側（DECISIONS.md）に置き、Wordからリンクする
- 決定は、出席している偉いプロパーの「異論なければ確定」サインオフを取って確定とする。覆されたときのために、経緯もDECISIONS.mdに残す
- 共有部分（数字の記録ルール・共通の土台コードなど）の変更は、PRで技術合意 → 結論を DECISIONS.md に1行。PR合意と決定ログを二重に管理せず、この順で1本化する
- Wordで決めた「やること」は、GitHubのIssueに落として実行に移す（橋渡しは翌朝の受託開発者MTGで）

## コスト・トークンの管理

予算上限と打ち切り閾値はチームの合意事項。初回MTGの議題として決め、DECISIONS.mdに残す（テックリードが事前に決めるのではない）。決めたらここに転記する。

- 予算上限（月あたりのAPI費用）: MTGで決める → （記入）
- 打ち切り閾値: 超えたら縮退（モデルを下げる・エージェント数を減らす・実験を止める）→ （記入）
- 運用: テックリードが朝会で今週の実測を見て、閾値に照らして手を打つ
- 実測は毎週、報告書のトークン欄に反映する（報告の数字を運用にも使う）

## 秘匿情報の扱い

- APIキー・トークンはコミットしない。環境変数か、Gitに載せない設定ファイルで持つ
- 実データ・個人情報はリポジトリに置かない。`.gitignore` で除外する
- 共有リポジトリ1つなので、各自がうっかり置かないよう最初に周知する

## やらないこと（短期PoCなので）

- CI整備・厳格なブランチ保護・細かいコミット規約に時間を使わない
- コードは動いて報告書に数字が載ればいい。作り込みすぎない
