# 精度改善 — 実測後の結論

本アプリ(出典参照つき RAG チャット)の回答精度を、評価ハーネスで measured に改善した記録の**結論版**。
試行錯誤の詳細ログは `scripts/eval/EXPERIMENTS.md`(EXP-001〜008)。
原則: **計測できないものは改善しない。集計値でなくトレースを見る。1変数ずつ A/B。**

> ⚠️ 当初の提案(日本語FTS/見出し埋め込み/map-reduce 等)は**ほとんど e2e を動かさなかった**。
> 実測で**真因は別の場所**だったと判明したため、効果の無かった評価は本文で**取消**として明記する。

---

## 0. 結論(最重要)

**真因 = 検索エージェントが合成(チャットエージェント)を不完全な文脈で飢えさせていた。**
- 1段目 `runSearchAgent` は agentic にツールを選ぶが、**ツール選択を誤り(keyword だけ呼ぶ等)候補プールに必要チャンクが入らない**、さらに `select_sources` で**少数(〜3個)しか選ばず**取りこぼす。
- 2段目 `streamAnswer`(合成)は渡されたチャンクから書くだけ。**合成自体は悪くない**。不完全な答えは不完全な文脈のせい。

**効いた改善(採用・デフォルト化済)**: 選抜に縛らず、**直接 `hybridSearch(question,12)` を足して網羅文脈を合成に渡す**(`agents.ts`、旧挙動は `EVAL_NARROW=1`)。
- EXP-008: e2e mean **1.385→1.520(+0.135)** / correct **53.4→60.7%** / partial+ **85→91%** / cited **83→95%**。全指標が一貫して上昇。
- 有意性 ~1.7σ(95%に僅か届かず)だが副指標が全部上で実効ほぼ確実。要・追試で確定。

---

## 1. 真因に至る診断(実測の経緯)

1. **検索(ファイル単位)は飽和していた**。JP docs で vector/hybrid とも Hit@10 97-99%。→ 「正しい資料を探す」はボトルネックでない。
2. **e2e は ~53%/mean1.38 から動かない**。retrieval をいじっても伝わらない/逆効果(後述の取消2件)。
3. **トレース(`EVAL_TRACE=1`)で実挙動を見て判明**:
   - 「確認できない」放棄(score0) = **答えのブロックが top-10 に無い**(検索ロス)。例: "12%" は対象ファイルに4チャンクあるが top-10 に0。
   - 部分回答(score1) = ブロックは取れるのに**検索エージェントが取りこぼす**。同一質問3回で 2/6→6/6→2/6 と**非決定的に必要チャンクを落とす**(e2e ノイズの正体でもある)。
   - さらに、候補プール自体が**エージェントのツール選択依存**で網羅性を欠く(keyword だけ呼んで vector を呼ばない等)。
4. → **検索エージェント→合成の文脈受け渡しが穴**。直接の網羅検索を足すと 6/6 が毎回揃い、e2e が上がった(EXP-008)。

**方法論の学び**: ①集計値(ファイル Hit)だけ見ると「合成が悪い」と誤断する。**block 単位＋実トレース**を見る。
②エージェント系の e2e は非決定的 → **k回平均＋95%CI**(`--runs`)でノイズ帯を出さないと delta を判定できない。

---

## 2. Open Notebook は不向きだった(忠実実装+paired 測定で反証)

最終的に ON を**忠実実装して厳密に測った**結論: **ON のアーキは本アプリのタスクに有意に劣る**。当初「ON を借りる」前提は誤りだった。

- **ON の実体**(`ask.py` 精読): 分解 → 各サブで vector_search → **各サブを“テキスト小回答”に圧縮(抽出)** → テキスト群を統合。
  **出典 id は最終回答へ流さない＝ON に出典機能は無い**(うちの看板=chunk_id→ハイライト と非互換)。
- **忠実実装 EVAL_ON の結果(EXP-013)**: paired vs wide **-0.313(有意に悪い)**、全構成中最下位。
  理由: テキスト抽出が **lossy**(数値・詳細が落ちる)。合成が**生チャンクに触れず薄い要約だけ**受け取り、詳細を回復できない。
- **試した派生も全て不発**: P1-3(chunk_id 抽出=けち, EXP-006 悪化) / DECOMP(抽出なし=ノイズ, EXP-010 wide 未満)。
- **効いた wide は ON の正反対**: 圧縮せず**生の網羅チャンクを出典付き合成に渡す**。これは ON 由来ではなく**自前トレース診断**(合成を文脈で飢えさせない)から出た。
- ON は広い要約/ポッドキャスト生成向き。**精密な事実 QA + 出典**には不向き、で確定。検索融合(RRF+hybrid)も元々うちが上。

## 4. 検証フロー(ハーネス・指標・方法論)

`scripts/eval/`:
- `run.ts` — 検索層 A/B(`--mode vector|keyword|hybrid`、決定的)。Hit@k/MRR。
- `e2e.ts` — 本番 `runSearchAgent → streamAnswer` を駆動し judge で `target_answer` と照合。
  **並列(`--concurrency`)/ k回平均(`--runs`)/ 平均スコア95%CI / 進捗 live(`results/.e2e-progress`)**。
- `score.ts` — Hit@k/MRR を domain/type 別に集計、baseline と delta。
- `gen-cases.ts` — 英語ゴールド生成 → `queries.en.json`。

**指標と原則**:
- **主役 = e2e(judge, mean±95%CI)**。ノイズ帯(N=50×3 で ±0.1 前後)を超えて初めて有意。
- **検索 Hit@k は決定的・安いので併走**(診断・天井)。ただし**ファイル単位は飽和して鈍い** → block 単位/トレースで見る。
- **トレース必須**: 各 A/B で chunk_id 差分(Jaccard)/ 問題ごと score 差分 / 比較ファイルの素性確認。集計だけで判断しない。
- ゴールド: `queries.json`(JP 234) と `queries.en.json`(EN cross-lingual 20)。JP はページ境界が md に無くファイル単位 Hit が天井。

**コーパス差(EXP-004/007)**:
- **日本語 docs**: 検索飽和(97-99%) / e2e 53% / 難問 → 弱点は文脈受け渡し。
- **英語 markdowns(cross-lingual)**: 検索**未飽和**(vector90/hybrid80) / e2e 76% / 易問 → 検索自体に伸びしろ。keyword は生の日本語Qで弱い(45%、agent は英語語に翻訳して補う)。
- → 改善の効きどころが JP/EN で違う。

## 5. 残課題と次の手

1. **EVAL_WIDE の有意性を追試で確定**(N拡大 or runs 増、安い)。
2. **アーキテクチャ簡素化の検討**: 網羅 hybridSearch が効くなら、**取りこぼしの元である agentic 検索ループ自体を簡素化/廃止**して `hybridSearch → 合成` 直結にできるか測る(速度・精度とも改善の可能性)。
3. **放棄ケース(score0=検索ロス)**: block が top-10 に入らない問題。チャンク粒度(断片・床併合)や ranking の別対応。
4. **ON の精読**: 抽出を**テキスト小回答**で返す ON 方式を、出典契約(chunk_id)を保ちつつ取り込めるか。
5. 英語側は**検索改善**(cross-lingual クエリ翻訳 / hybrid で弱 keyword を down-weight)。

## 6. 評価の天井 = gold の質(重要・2026-06 追記)

**真のボトルネックは検索でも合成でもなく、`queries.json` の gold(target_answer)の質。**
gold は自動生成で、**アプリの回答より不正確なことがある**。judge は gold を正解基準に採点する(`e2e.ts`: 「検索資料は与えられないので模範解答を正解の基準とする」)ため、**アプリが gold より正確だと、むしろ減点され得る**。

実例(質問: 2017-2021 で自然災害の保険金支払が最大の年と理由):
- **gold**: 「2018年度、**10,678億円**の被害。理由は台風21号」← 台風21号*単体*の額を年総額と混同。
- **アプリ**: 「2018年度、年計**1.60兆円**。うち台風21号が**1兆678億円**」← 年総額と単一イベントを正しく分離。**アプリの方が正確**。

含意:
- e2e の「正答率 ~60-70%」は**真の品質を過小評価**している可能性が高い。
- 改善が gold と食い違う方向だと減点される → **gold ノイズ(±0.1級)未満の改善は原理的に検出不能**。検索チューニング(下記)が軒並み e2e フラットだった真因はこれ。

→ 次に精度を「上げる/測る」なら、手を入れるべきは**検索ロジックではなく評価系**:
- 怪しい target_answer を人手で正す、またはより厳密なデータセットへ。
- judge を「gold 厳密一致」ではなく「**原資料に照らした事実性**」採点へ変える(資料を judge に見せる)。
